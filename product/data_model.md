# Data Model (Conceptual)

Not SQL yet — this describes entities, their attributes, and relationships.

---

## Entities

### User
- id
- username (unique, displayable)
- avatar_url
- created_at
- home_city (optional, for regional defaults)
- xp (total experience points)
- level (derived from xp)

### Bird (Species)
- id
- common_name (localized: DE, EN)
- scientific_name
- family
- description
- photo_url
- audio_sample_url (song/call)
- typical_regions [ ]
- typical_months [ ] (when it's present / active)
- rarity_score (common / uncommon / rare / very rare) — per region
- fun_facts [ ]

### Catch
- id
- user_id → User
- bird_id → Bird
- caught_at (timestamp)
- location (lat, lng) — PostGIS Point
- method: enum(naked_eye, binoculars, heard_only, photo, sound_id)
- note (optional text)
- photo_url (optional)
- visibility: enum(public, friends, private)
- is_first_for_user (boolean — "new species" flag)

### Friendship
- id
- requester_id → User
- addressee_id → User
- status: enum(pending, accepted, blocked)
- created_at

### Badge
- id
- key (unique slug, e.g. "first_catch", "woodpecker_family")
- name
- description
- icon_url
- trigger_condition (description of how it's earned)

### UserBadge
- id
- user_id → User
- badge_id → Badge
- earned_at

### Notification
- id
- user_id → User (recipient)
- type: enum(rare_bird_nearby, friend_catch, new_follower, weekly_recap, milestone)
- payload (JSON — flexible per type)
- read_at (nullable)
- created_at

---

## Key Relationships

```
User ──< Catch >── Bird        (many-to-many through Catch)
User ──< Friendship >── User   (self-referential many-to-many)
User ──< UserBadge >── Badge   (many-to-many through UserBadge)
User ──< Notification          (one-to-many)
```

---

## Key Queries (to design for)

1. "All public catches of species X near location Y within radius Z" → heatmap, map pins
2. "All catches by user X" → profile / collection
3. "All catches by friends of user X in last 7 days" → activity feed
4. "Has user X ever caught species Y?" → new species check
5. "Hottest species in city Z this month" → trending
6. "All catches within bounding box" → map viewport loading

---

## Notes for DB design

- Catches need PostGIS geography type for efficient spatial queries
- `Bird` table should be seeded from an authoritative source (e.g. eBird taxonomy, xeno-canto)
- Rarity is region-dependent — may need a `BirdRegionRarity` join table later
- Notifications should be generated by DB triggers or edge functions, not in app logic
